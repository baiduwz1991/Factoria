# Vibe Coding 工厂游戏设计构想

## 核心概念

把玩家建造的工厂设计成一个可以被 AI 理解、分析、修改和验证的“工程项目”。

玩家不是单纯手动摆放建筑，而是像在 Cursor 里维护代码工程一样维护一个复杂工厂系统：

- 工厂地图 = 代码仓库
- 蓝图区域 = 模块 / 文件
- 生产线 = 函数逻辑
- 输入输出口 = API
- 物流、电力、管线 = 依赖关系
- 堵塞、空转、断电 = bug
- 产能不足 = 性能瓶颈
- 优化方案 = refactor
- 模拟运行 = test
- 蓝图版本 = commit / diff / revert

一句话概括：

> 让玩家在工厂世界里体验“选中一段复杂系统，然后让 AI 读懂、修改、测试、提交”的快感。

## Vibe Coding 的爽感来源

Vibe Coding 的爽感不只是“输入一句话生成东西”，而是让玩家获得一种新的控制力：

- 解决繁杂任务
- 处理自己不想手动完成的大量细节
- 借助 AI 完成知识面过广、个人能力难以覆盖的任务
- 从执行者变成架构师
- 通过描述意图、审查方案、运行测试、持续迭代来掌控复杂系统

因此游戏本体应该足够庞杂。复杂度不应该来自繁琐操作，而应该来自真实系统之间的耦合、瓶颈、冲突和长期演化。

## 参考对象：《异星工场》

《异星工场》非常适合作为底层参考，因为它已经证明了：

- 庞杂系统本身有吸引力
- 自动化带来强烈成就感
- 瓶颈排查和系统重构很有乐趣
- 蓝图复用能形成长期积累

但传统《异星工场》更像“亲手写代码”。

本游戏希望进一步变成“意图驱动工程”：玩家不是亲自移动每一条传送带，而是向 AI 描述目标，让 AI 分析、生成、修改蓝图，玩家负责判断、验收和继续调教。

## 典型玩法循环

1. 玩家建造或框选一块工厂区域。
2. 玩家向 AI 提问或下达目标。
3. AI 分析当前结构，给出诊断。
4. AI 提供一个或多个优化方案。
5. 玩家查看类似 Cursor diff 的改造预览。
6. 玩家选择接受、部分接受、重新生成或增加约束。
7. 系统运行模拟测试。
8. 玩家根据测试结果继续迭代。

这个循环对应真实 Vibe Coding 的工作流：

> 描述意图 -> 生成方案 -> 审查 diff -> 应用修改 -> 跑测试 -> 继续修正。

## 核心功能

### 1. 框选问诊

玩家可以框选一块区域并询问：

- 这里为什么产量上不去？
- 有没有优化空间？
- 哪些机器在空转？
- 哪条传送带是瓶颈？
- 是否存在死锁风险？
- 电力、物流、输入输出是否稳定？
- 这块区域是否适合继续扩展？

AI 输出诊断报告，包括：

- 当前输入
- 当前输出
- 理论产能
- 实际产能
- 空转率
- 堵塞点
- 电力消耗
- 空间利用率
- 扩展风险

### 2. AI 修改蓝图

玩家可以提出目标：

- 保持产量不变，把占地缩小 20%
- 不改变输入输出位置，重排内部结构
- 把产能提升到两倍
- 不使用高级传送带
- 优先降低电力消耗
- 预留后续扩展空间
- 不要动左侧主线
- 把所有输入口放在左边，输出口放在右边

AI 根据目标修改蓝图，并生成可预览的改造方案。

### 3. Diff 预览

AI 修改前，游戏展示类似 Cursor 的差异预览：

- 红色表示拆除
- 绿色表示新增
- 黄色表示移动或替换
- 箭头表示新的物流路径
- 指标面板显示修改前后变化

玩家可以选择：

- 接受全部
- 只接受部分改动
- 让 AI 重新生成
- 增加约束后再试
- 放弃修改

### 4. 多方案对比

AI 不只给一个答案，而是给多个工程方案：

- 低成本修补方案
- 高产能重构方案
- 长期可扩展方案
- 节能方案
- 最小改动方案

玩家根据当前资源、时间、风险和战略目标选择方向。

### 5. 模拟测试

AI 改完后不能默认正确，必须经过验证。

系统可以运行一段模拟，检查：

- 是否达到目标产量
- 是否出现堵塞
- 是否有机器长期空转
- 是否存在资源周期性断供
- 是否引发外部主线拥堵
- 是否消耗过多电力
- 是否破坏周边结构

这保留了调试乐趣，也让 AI 修改不是“一键通关”。

### 6. 版本管理

每次 AI 修改都生成版本记录：

- 修改说明
- 改动范围
- 修改前后指标
- 风险说明
- 可回滚版本

玩家可以像维护工程一样管理自己的工厂演进。

## AI 角色定位

AI 不应该是“替玩家玩游戏”，而应该是“工程助手”。

AI 负责：

- 分析复杂结构
- 执行繁琐改造
- 生成候选方案
- 解释系统瓶颈
- 帮玩家把目标转化为可执行修改

玩家负责：

- 提出目标
- 设置约束
- 判断取舍
- 审查方案
- 选择是否应用
- 根据测试结果继续迭代

这样玩家不是失去参与感，而是从工人升级为架构师。

## 技术实现思路

这个玩法的关键不是单纯接入大模型，而是让工厂系统具备以下能力：

- 结构化表示
- 可分析
- 可验证
- 可重放
- 可增量修改
- 可生成 diff
- 可回滚

建议拆成三层。

### 模拟器

负责真实运行工厂，输出数据：

- 产量
- 消耗
- 堵塞
- 空转
- 电力
- 仓储
- 物流延迟
- 资源波动

### 优化器

负责根据目标生成可行修改：

- 提高产量
- 压缩空间
- 降低能耗
- 保持接口不变
- 降低材料成本
- 提升扩展性

### AI 助手

负责理解玩家语言、调用内部工具、解释结果，并把结构化修改转化为玩家能理解的方案。

AI 最好不是直接凭空生成答案，而是基于游戏内部工具进行分析和改造。

## 最小可行原型

优先实现一个小闭环：

> 玩家框选一块生产区域，AI 给出诊断报告，并提供 1-3 个可应用优化方案。

原型只需要支持：

- 选区读取
- 输入输出统计
- 机器空转统计
- 简单瓶颈判断
- 简单布局改造
- 修改前后对比
- 应用或回滚

只要这个闭环成立，核心体验就成立。

## 设计目标

最终目标不是做一个“AI 自动玩工厂”的游戏，而是：

> 做一个让玩家用 AI 驾驭复杂工程系统的游戏。

玩家获得的不是偷懒感，而是掌控感：

- 我能理解复杂系统
- 我能改造复杂系统
- 我能让 AI 处理繁杂细节
- 我能通过目标、约束和测试持续推进工程
- 我一个人也能维护原本需要团队才能完成的庞大系统
