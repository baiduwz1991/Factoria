# Agent 蓝图玩法设计

## 设计定位

Agent 蓝图系统是 FACTORIA 区别于传统工厂游戏的核心能力。

它把蓝图从“静态复制粘贴数据”升级为“可被自然语言生成、解释、修改和优化的工程对象”。玩家不需要一次性手动摆完所有细节，可以通过和 agent 对话逐步得到可用方案。

## 核心循环

1. 玩家提出意图或框选区域。
2. 系统收集上下文：地图、建筑、资源、约束、已有蓝图或选区内容。
3. agent 生成分析、方案或修改建议。
4. 游戏把结果转成可预览的蓝图草案。
5. 玩家确认、继续修改或放弃。
6. 确认后蓝图才进入世界或蓝图库。

## 三类主要能力

### 1. 自然语言生成蓝图

玩家输入目标，agent 产出蓝图草案。

示例输入：

> 做一个早期铁板产线，输入铁矿，输出铁板，适合前期扩建。

agent 需要理解：

- 目标产物。
- 输入资源。
- 产能倾向。
- 布局偏好。
- 可用建筑。
- 玩家当前科技阶段。

输出应包含：

- 蓝图数据。
- 建筑清单。
- 输入输出接口说明。
- 设计解释。
- 未满足约束或假设条件。

### 2. 修改已有蓝图

玩家选择一份蓝图后，用自然语言要求修改。

示例输入：

> 把它改成左右对称，并把输出口放到右侧。

常见修改类型：

- 调整方向。
- 放大产能。
- 压缩占地。
- 改变输入输出位置。
- 替换设备等级。
- 增加缓冲或冗余。
- 拆分成多个模块。

修改过程应尽量以差异形式呈现，让玩家知道 agent 改了什么。

### 3. 框选区域分析

玩家在地图上框选一块已有工厂区域，agent 读取区域内的实体和连接关系。

示例输入：

> 分析这个区域的瓶颈，并给我一个优化方案。

系统需要从选区提取：

- 建筑列表。
- 建筑位置和朝向。
- 传送带、管道、电力等连接关系。
- 输入输出边界。
- 当前生产链路。
- 可观测的缺口或拥塞。

agent 输出：

- 区域功能摘要。
- 关键生产链路。
- 瓶颈判断。
- 可执行优化建议。
- 是否可转成蓝图。

## 蓝图对象边界

蓝图应先设计成可被程序和 agent 共同理解的结构化数据，而不是只依赖场景节点。

建议蓝图至少表达：

- 实体类型。
- 网格坐标。
- 朝向。
- 配方或工作模式。
- 连接关系。
- 输入输出端口。
- 占地范围。
- 版本号。
- 可选元数据：名称、说明、标签、作者、agent 备注。

早期可以先忽略复杂运行时状态，只保留建造所需信息。

## Agent 输出约束

agent 不能直接绕过游戏规则。

生成或修改蓝图时必须经过校验：

- 建筑是否已解锁。
- 建筑是否存在。
- 坐标是否冲突。
- 连接是否可达。
- 输入输出方向是否合理。
- 占地是否超出限制。
- 资源成本是否可计算。

如果校验失败，系统应展示原因，并允许 agent 基于错误继续修正。

## UI 交互草案

### 蓝图对话入口

可从蓝图库、地图框选工具或快捷入口打开。

基础区域：

- 对话输入框。
- 当前上下文摘要。
- 蓝图预览。
- 变更说明。
- 确认应用按钮。
- 继续修改按钮。

### 预览与确认

agent 结果默认进入预览态，而不是直接写入地图。

预览态应支持：

- 显示蓝图占地。
- 标记输入输出口。
- 高亮新增、删除、调整的实体。
- 展示资源成本。
- 展示校验错误。

玩家确认后才真正放置或保存。

## 最小可行版本

第一阶段不需要完整智能体。

可以先做一个规则驱动或本地模拟的 agent 流程：

1. 定义蓝图数据结构。
2. 实现蓝图序列化和反序列化。
3. 实现地图框选到蓝图数据。
4. 实现蓝图预览与放置。
5. 做一个简单文本指令入口，支持少量模板指令。
6. 为未来 LLM 接入预留统一的请求与响应格式。

这样可以先验证“玩家和蓝图对话”是否好玩，再决定 agent 智能程度。

## 后续待设计问题

- agent 是否消耗游戏内资源或时间。
- agent 生成蓝图是否受玩家科技等级限制。
- 蓝图分析是否需要真实产能模拟。
- 如何表示“不确定性”和“假设条件”。
- 如何避免玩家只让 agent 自动最优解而失去游戏乐趣。
- 是否提供 agent 性格、工程风格或专长差异。
- 是否支持多人共享 agent 生成的蓝图。
